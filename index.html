    Start an AIP #

    Begin by checking whether your idea is already covered by an existing AIP. If it is new, write the idea clearly enough for editors, developers, and the community to review. A strong draft explains the problem, the proposed change, the expected implementation work, and the parties affected by the change.

    1. Review the current AIPs and confirm that the proposal is not a duplicate.
    2. Discuss the idea publicly and collect feedback before preparing a formal draft.
    3. Find co-authors when the proposal needs domain, implementation, or governance support.
    4. Move from notes to a Markdown AIP draft when the idea is ready for review.

    Submit, Update, and Finalize #

    Formal AIP changes happen through the AIPs repository. A first submission becomes a draft after review, draft updates require author participation, and mature drafts can request final or active status.

    First submission

    Fork the AIPs repository, add the draft in the correct AIP directory, change the status to Draft when it is ready, then open a pull request. Editors review the draft and assign the AIP number.

    Draft update

    Only listed authors should update a draft AIP. The pull request should explain what changed since the last draft and keep discussion links available for reviewers.

    Final or active request

    When a draft is mature, authors can request finalization. Editors and the review board check consensus, implementation readiness, and whether a public review period is needed.

    Format and Templates #

    An AIP is written in Markdown and stored as an index.md file inside its AIP directory. Images and supporting files should stay in the same directory and be linked with relative paths. Browser-executable files such as HTML or JavaScript are not accepted as proposal assets.

    Required fieldPurpose
    AIPThe number assigned by the editor.
    TitleThe quoted proposal title.
    AuthorsThe author names, GitHub usernames, or public emails.
    DiscussionsA public discussion URL while the AIP is still a draft.
    StatusThe current lifecycle state, such as WIP, Draft, Public Call, Final, or Implemented.
    CategoriesThe proposal category, including Technical or Community Governance for the current site.
    CreatedThe first creation date in YYYY-MM-DD format.
    TypesThe proposal type, such as Standard, ARC, or Informational.

    AIP Process #

    A normal AIP moves from early writing to editor review, public review, final acceptance, and implementation. Authors request lifecycle changes through repository updates, while editors and the review board check quality, consensus, and implementation readiness.

    WIP Draft Public Call Final or Active Implemented

    Special process paths can also happen. An active AIP can receive feature updates, an abandoned AIP can be picked up by a new champion, and a final AIP can later be superseded by a replacement proposal.

    AIP Status #

    Status values explain where a proposal sits in the lifecycle and what kind of action is still allowed.

    • WIP: The author is still shaping the idea before repository submission.
    • Draft: The first version has entered the AIP repository and can still receive updates.
    • Public Call: The review board invites final public checks before finalization.
    • Final or Active: The proposal is accepted. Final AIPs are no longer changed, while active AIPs can accept feature updates.
    • Implemented: The accepted proposal has been implemented.
    • Deferred, Superseded, Abandoned, or Rejected: The proposal is paused, replaced, no longer maintained, or not accepted.

    Ownership and Maintenance #

    An AIP may need a new champion if the original author is unavailable or no longer maintaining it. Ownership transfer should preserve continuity, keep the proposal reviewable, and make clear who is responsible for future updates.

    1. Try to contact the listed authors first.
    2. If authors cannot be reached, ask AIP editors for assistance.
    3. Editors may mark abandoned work accordingly and allow a new champion to continue it.
    4. The new owner should document the reason for transfer and keep future updates scoped.
